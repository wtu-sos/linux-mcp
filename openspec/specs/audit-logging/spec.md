## 新增需求

### 需求：记录所有操作到审计日志
系统必须将每个工具操作记录到远程主机上的审计日志文件。

#### 场景：成功操作记录
- **当** 任何 MCP 工具成功完成
- **则** 系统向 `~/.linux_mcp_audit.log` 追加一行 JSON，包含时间戳、操作名称、参数、结果状态和退出码

#### 场景：失败操作记录
- **当** 任何 MCP 工具执行失败
- **则** 系统追加一行 JSON，结果状态为 "error" 并包含错误信息

#### 场景：被拒绝操作记录
- **当** 操作被安全网关拒绝（例如危险命令无回滚方案）
- **则** 系统追加一行 JSON，结果状态为 "rejected" 并包含拒绝原因

### 需求：审计日志记录格式
系统必须使用一致的 JSON Lines 格式记录审计条目。

#### 场景：标准记录字段
- **当** 记录操作时
- **则** 记录包含：`timestamp`（ISO 8601）、`operation`（工具名称）、`params`（关键参数）、`result`（success/error/rejected）、`exit_code`（如适用）、`rollback_id`（如适用）、`error`（如失败）

### 需求：查询审计日志
系统必须提供读取和过滤审计日志的工具。

#### 场景：读取最近审计条目
- **当** 调用 `get_audit_log` 且无过滤条件
- **则** 系统返回审计日志中最近 N 条记录（默认 50 条）

#### 场景：按操作类型过滤
- **当** 调用 `get_audit_log` 并指定操作类型过滤（例如 `delete_file`）
- **则** 系统仅返回匹配该操作类型的条目

#### 场景：按时间范围过滤
- **当** 调用 `get_audit_log` 并指定 `since` 和 `until` 时间戳
- **则** 系统仅返回该时间范围内的条目

### 需求：审计日志轮转
系统必须在审计日志超过配置的最大大小时进行轮转。

#### 场景：日志超出最大大小
- **当** 审计日志文件超过配置的最大大小（默认 10MB）
- **则** 系统将当前日志重命名为 `~/.linux_mcp_audit.log.1` 并开始新的日志文件
